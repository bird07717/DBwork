# 城市公交客流分析与智能调度 Agent 平台任务拆分 v1

## 1. 总体说明

本任务拆分覆盖阶段 1 到阶段 6。阶段 1 到阶段 6 完成后，项目应达到可本地运行、可联调、可演示的状态。

阶段 7 由用户自行进行测试、截图、验收和课程报告补充。

执行每个任务时必须遵守 `docs/execution-workflow.md`。

## 2. 阶段 1：项目初始化与基础设施

目标：建立可开发的 Monorepo 项目结构，并让 MySQL、Redis 通过 Docker Compose 运行。

### T1.1 创建项目目录结构

任务内容：

- 创建 `frontend/`
- 创建 `backend/`
- 创建 `ai-service/`
- 创建 `sql/`
- 创建 `scripts/`
- 保留已有 `docs/` 和 `data/`

完成标准：

- 根目录结构符合 `docs/mvp-v1.md` 中的 Monorepo 结构。
- 各模块目录职责清晰。

检查验证：

- 使用目录列表检查目录是否存在。

### T1.2 创建 Docker Compose 基础设施

任务内容：

- 创建 `docker-compose.yml`
- 配置 MySQL 8.0
- 配置 Redis
- 配置数据库名、端口、账号、密码和数据卷
- 配置 SQL 初始化挂载目录

完成标准：

- `docker compose up -d` 可以启动 MySQL 和 Redis。
- MySQL 和 Redis 容器状态正常。

检查验证：

- 运行 `docker compose config`。
- 运行 `docker compose up -d`。
- 检查容器状态。

### T1.3 创建环境变量示例

任务内容：

- 创建前端 `.env.example`
- 创建后端配置模板或说明
- 创建 AI 服务 `.env.example`
- 写入 `VITE_TENCENT_MAP_KEY`、数据库连接、大模型 API Key 等变量名

完成标准：

- 不在示例文件中写入敏感生产密钥。
- 腾讯地图 Key 的使用方式与 `docs/map-integration.md` 一致。

检查验证：

- 检查变量名是否与文档一致。

### T1.4 创建 README 初版

任务内容：

- 创建 `README.md`
- 写明项目定位、技术栈、目录结构和基础启动顺序

完成标准：

- README 能让开发者理解项目结构和第一步启动方式。

检查验证：

- 人工阅读 README，确认没有与需求文档冲突。

### 阶段 1 结束检查

必须验证：

- 目录结构完整。
- Docker Compose 配置有效。
- MySQL 和 Redis 可启动。
- 环境变量示例存在。
- README 初版存在。
- 文档已同步阶段状态。

通过后进入阶段 2。

## 3. 阶段 2：数据库落地与数据生成

目标：将数据库设计落成 SQL，导入 3 条真实线路，生成第一版模拟运营数据。

### T2.1 编写 `sql/schema.sql`

任务内容：

- 根据 `docs/database-design.md` 创建表结构。
- 包含 `admin_user`、`bus_route`、`station`、`route_station`、`bus_vehicle`、`driver`、`bus_schedule`、`ride_record`、`dispatch_advice`、`operation_log`。
- 添加主键、外键、唯一约束和索引。

完成标准：

- SQL 可以在 MySQL 8.0 执行成功。
- 表结构与数据库设计文档一致。

检查验证：

- 在 Docker MySQL 中执行 `schema.sql`。
- 检查所有表是否创建成功。
- 检查关键索引和外键是否存在。

### T2.2 编写管理员初始化 SQL

任务内容：

- 创建 `sql/init_admin.sql`。
- 初始化管理员账号，显示姓名为课程项目管理员。
- 初始化密码不得明文入库，必须写入加密后的 `password_hash`。

完成标准：

- 管理员账号能用于后端登录。
- 数据库中不出现明文密码。

检查验证：

- 查询 `admin_user` 表。
- 确认 `password_hash` 不是明文密码。

### T2.3 编写真实线路导入脚本

任务内容：

- 创建 `scripts/import_route_data.py`。
- 读取 `data/real_routes/route_1.json`、`route_2.json`、`route_3.json`。
- 写入 `bus_route`、`station`、`route_station`。
- 同名同坐标站点需要复用，避免重复站点。

完成标准：

- 3 条线路成功导入。
- 线路站点顺序正确。
- 共享站点在 `station` 表中合理复用。

检查验证：

- 统计线路数为 3。
- 检查 3 条线路站点数量分别为 27、17、32。
- 检查 `route_station.station_order` 连续且正确。

### T2.4 编写模拟数据生成脚本

任务内容：

- 创建 `scripts/generate_mock_data.py`。
- 生成 30 辆车，默认容量 50 人。
- 生成 30 名司机。
- 生成 30 天班次，每天约 100 条。
- 生成约 1 万条乘车记录。
- 按早高峰、晚高峰、正常时段模拟客流差异。

完成标准：

- 模拟数据规模符合 MVP 要求。
- 乘车记录能支撑统计图表和 AI 分析。

检查验证：

- 查询车辆数、司机数、班次数、乘车记录数。
- 检查 `ride_record.period_type` 分布。
- 检查乘车记录关联的线路和站点有效。

### T2.5 验证核心统计 SQL

任务内容：

- 验证线路日客流趋势 SQL。
- 验证站点客流 TOP SQL。
- 验证早晚高峰对比 SQL。
- 验证满载率计算 SQL。

完成标准：

- 复杂 SQL 可以返回合理结果。
- 查询结果能用于后端统计接口。

检查验证：

- 执行 `docs/database-design.md` 中列出的复杂 SQL。
- 记录查询结果是否正常。

### 阶段 2 结束检查

必须验证：

- MySQL 表结构可正常创建。
- 管理员初始化数据存在且密码非明文。
- 3 条真实线路导入正确。
- 模拟数据规模达标。
- 核心统计 SQL 返回合理结果。
- 文档已同步阶段状态。

通过后进入阶段 3。

## 4. 阶段 3：Java 后端开发

目标：完成主业务后端，提供游客公开接口、管理员后台接口、统计接口和 AI 服务调用能力。

### T3.1 创建 Spring Boot 项目

任务内容：

- 在 `backend/` 创建 Spring Boot 项目。
- 配置 Maven。
- 配置 MyBatis Plus。
- 配置 MySQL 和 Redis 连接。
- 配置 Knife4j。

完成标准：

- 后端项目可以启动。
- 能连接 Docker MySQL 和 Redis。

检查验证：

- 运行后端启动命令。
- 检查健康接口或启动日志。

### T3.2 实现通用基础能力

任务内容：

- 统一响应结构。
- 统一异常处理。
- 分页请求和分页响应。
- 基础时间格式处理。

完成标准：

- 后续接口统一返回结构。
- 异常能返回可读错误信息。

检查验证：

- 编写或调用测试接口验证响应格式。

### T3.3 实现管理员登录鉴权

任务内容：

- 使用 Sa-Token 实现管理员登录。
- 校验初始化管理员账号。
- 后台接口需要登录态。
- 公开接口不需要登录。

完成标准：

- 管理员可以登录。
- 未登录访问后台接口被拒绝。
- 游客接口可正常访问。

检查验证：

- 测试登录成功、登录失败、未登录访问后台接口。

### T3.4 实现公开线路和地图接口

任务内容：

- `GET /api/public/routes`
- `GET /api/public/routes/{id}`
- `GET /api/public/routes/{id}/stations`

完成标准：

- 前端能拿到 3 条线路。
- 线路详情包含站点顺序和经纬度。

检查验证：

- 调用接口确认返回数据。
- 检查站点顺序和 JSON 数据一致。

### T3.5 实现后台基础数据管理接口

任务内容：

- 线路 CRUD。
- 站点 CRUD。
- 车辆 CRUD。
- 司机 CRUD。
- 班次 CRUD。
- 乘车记录查询。

完成标准：

- 基础数据管理页可以调用接口完成列表、新增、修改、删除或停用。

检查验证：

- 使用接口测试工具或自动测试验证 CRUD。
- 检查外键约束下的删除行为。

### T3.6 实现客流统计接口

任务内容：

- 线路日客流趋势。
- 站点客流排行。
- 早晚高峰和正常时段对比。
- 满载率统计。

完成标准：

- 客流分析页可以获取图表数据。
- AI 分析可以获取结构化统计数据。

检查验证：

- 调用统计接口。
- 对比 SQL 查询结果。

### T3.7 实现调度建议接口

任务内容：

- 根据线路、时间范围、时段类型查询统计数据。
- 使用默认阈值生成规则结果。
- 调用 Python AI 服务生成自然语言说明。
- 保存 `dispatch_advice`。

完成标准：

- 管理员可以生成并保存调度建议。
- 返回内容包含关键指标和建议说明。

检查验证：

- 测试高峰、正常时段两类建议。
- 检查数据库保存记录。

### T3.8 实现 AI 服务客户端

任务内容：

- 后端封装调用 Python FastAPI 的 HTTP 客户端。
- 处理 AI 服务超时、异常和降级。

完成标准：

- AI 服务不可用时后端能返回规则建议或明确错误提示。

检查验证：

- AI 服务正常时调用成功。
- AI 服务停止时后端不崩溃。

### 阶段 3 结束检查

必须验证：

- 后端可启动。
- 管理员登录正常。
- 游客公开接口正常。
- 后台基础数据接口正常。
- 统计接口正常。
- 调度建议接口可调用或具备降级逻辑。
- Knife4j 接口文档可访问。
- 文档已同步阶段状态。

通过后进入阶段 4。

## 5. 阶段 4：Python AI 微服务开发

目标：实现客流分析和调度建议两个 AI 能力，供 Java 后端调用。

### T4.1 创建 FastAPI 项目

任务内容：

- 在 `ai-service/` 创建 FastAPI 项目。
- 配置依赖文件。
- 配置 `.env.example`。
- 提供健康检查接口。

完成标准：

- AI 服务可以本地启动。
- 健康检查接口可访问。

检查验证：

- 运行 `uvicorn app.main:app --reload`。
- 调用健康检查接口。

### T4.2 定义 AI 输入输出 Schema

任务内容：

- 定义客流分析请求和响应。
- 定义调度建议请求和响应。
- 明确字段包含线路、时间范围、统计数据、建议结果。

完成标准：

- Java 后端和 AI 服务的 JSON 结构一致。

检查验证：

- 使用样例 JSON 调用接口。

### T4.3 实现客流分析接口

任务内容：

- 实现 `POST /ai/analyze-flow`。
- 基于后端传入统计数据生成分析报告。
- 不直接查询数据库。

完成标准：

- 返回分析摘要、关键数据说明和客流变化判断。

检查验证：

- 使用模拟统计数据调用接口。
- 检查回答是否基于输入数据。

### T4.4 实现调度建议接口

任务内容：

- 实现 `POST /ai/dispatch-advice`。
- 接收规则计算结果和统计数据。
- 输出调度建议、原因、风险提示、优先级。

完成标准：

- 能区分高峰模板和正常时段模板。
- 不生成与输入数据冲突的建议。

检查验证：

- 使用高峰样例和正常时段样例分别测试。

### T4.5 实现 AI 降级策略

任务内容：

- 大模型 API Key 未配置时返回基于模板的分析。
- 大模型调用失败时返回可解释的降级结果。

完成标准：

- 没有大模型 Key 时项目仍可演示基本 AI/规则分析。

检查验证：

- 不配置大模型 Key 启动服务。
- 调用接口确认不会崩溃。

### 阶段 4 结束检查

必须验证：

- AI 服务可启动。
- 健康检查正常。
- `/ai/analyze-flow` 正常。
- `/ai/dispatch-advice` 正常。
- Java 后端可调用 AI 服务。
- 大模型不可用时有降级能力。
- 文档已同步阶段状态。

通过后进入阶段 5。

## 7. 阶段 6：前后端与 AI 联调

目标：完成端到端联调，使项目达到可运行、可演示状态。

### T6.1 联调游客线路与地图流程

任务内容：

- 首页调用公开线路接口。

任务内容：

完成标准：

- 游客能完成一次自然语言客流分析。

### T6.4 联调管理员后台流程

任务内容：

完成标准：

- 管理员核心流程可用。

检查验证：

### T6.5 联调异常和降级流程

任务内容：

- 测试 AI 服务不可用。
- 测试地图 Key 缺失。
- 测试后端未登录访问后台接口。

完成标准：

- 系统有明确错误提示或降级处理。

检查验证：

- 手动关闭 AI 服务测试。
- 清空或切换错误配置测试。

### T6.6 可运行交付检查

任务内容：

- 整理启动顺序。
- 确认 Docker、后端、AI 服务、前端均能启动。
