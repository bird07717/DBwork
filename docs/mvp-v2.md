# 城市公交客流分析与智能调度 Agent 平台 MVP v2

## 1. 版本定位

MVP v2 定位为：**答辩展示 + 简历技术亮点增强版**。

第一版已经完成从 0 到 1 的可运行闭环：游客端、管理员端、地图、客流统计、AI 分析、调度建议、MySQL 数据库、Redis、Java 后端、Python AI 微服务和 Vue 前端均已具备本地运行与演示能力。

第二版不再追求大范围新增业务模块，而是围绕“展示效果更强、技术亮点更清楚、答辩讲解更顺畅、简历描述更有含金量”进行增强。

本版本重点优化四个方向：

1. 首页态势展示再增强。
2. 客流分析页增强。
3. AI 助手报告化。
4. 后台调度建议增强。

## 2. 版本目标

### 2.1 答辩展示目标

MVP v2 应让答辩时的演示链路更有冲击力：

- 打开首页后，第一眼能看出这是“公交运行态势平台”，而不是普通管理系统。
- 能通过地图、线路状态、站点压力、告警摘要和今日客流快速说明系统价值。
- 客流分析页能用图表清楚说明“高峰更拥挤、不同线路客流不同、热点站点存在”。
- AI 助手输出应像一份调度分析报告，方便截图和讲解。
- 管理员调度建议应展示“建议依据”，体现不是凭空生成。

### 2.2 简历技术亮点目标

MVP v2 应让简历描述更有内容：

- 前端：地图驱动大屏、图表分析、状态可视化、异常兜底、组件化展示。
- 后端：统计接口、调度建议生成、统一数据口径、AI 服务调用与降级。
- 数据库：线路、站点、班次、乘车记录建模，复杂统计查询，索引支撑分析。
- AI：基于结构化统计数据生成自然语言报告，规则建议与 AI 解读结合。
- 工程：前后端分离、Java + Python 微服务协作、Docker 基础设施、本地可演示。

## 3. 版本边界

### 3.1 本版本要做

- 强化现有页面表达，不从零重写系统。
- 优先复用现有接口和现有数据结构。
- 优先通过前端聚合、展示优化、AI 输出结构化增强效果。
- 必要时可新增少量后端统计接口，但应保持响应格式清晰、兼容现有功能。
- 完成后补充演示流程和验收清单。

### 3.2 本版本不做

- 不做真实车辆 GPS 定位。
- 不做乘客端注册登录。
- 不做司机端 App。
- 不接入真实支付或刷卡系统。
- 不做复杂权限角色体系。
- 不做生产服务器部署。
- 不大改数据库 schema，除非后续明确需要。
- 不引入大型新依赖。
- 不把系统改成纯大屏项目，仍保留游客端和管理员端完整链路。

## 4. 总体演示链路

MVP v2 的推荐答辩演示链路如下：

```text
打开首页态势看板
→ 展示今日客流、线路状态、站点压力、告警摘要
→ 切换不同线路，观察地图站点颜色和线路状态变化
→ 进入客流分析页，展示高峰/平峰对比、线路对比、站点排行
→ 进入 AI 助手页，选择线路和日期，生成调度分析报告
→ 管理员登录后台，进入调度建议页
→ 选择线路、日期、时段，生成调度建议
→ 展示满载率、客流量、热点站点、建议级别和建议依据
```

这个链路对应答辩讲解重点：

1. 系统能采集和管理公交运营数据。
2. 系统能用地图和图表分析客流情况。
3. 系统能基于客流统计生成调度建议。
4. 系统能通过 AI 把结构化数据转成可读报告。
5. 数据库设计支撑了线路、站点、班次、乘车记录和调度建议的完整闭环。

## 5. 功能优化点一：首页态势展示再增强

### 5.1 优化目标

首页应成为项目第一眼亮点。用户打开系统后，应立即看到公交运行态势，而不是普通入口页。

首页需要更清楚展示：

- 线路整体状态。
- 今日客流。
- 站点压力。
- 告警摘要。
- 当前选中线路的运行情况。

### 5.2 建议页面结构

首页建议采用“地图主视觉 + 左侧态势浮层 + 右侧线路站点面板 + 顶部总览指标”的结构。

```text
┌──────────────────────────────────────────────┐
│ 顶部总览：线路数 / 今日客流 / 最高满载率 / 告警数 │
├──────────────────────────────────────────────┤
│                                              │
│  ┌──────────────┐            ┌────────────┐  │
│  │ 左侧态势浮层   │            │ 右侧线路面板 │  │
│  │ 当前线路指标   │            │ 线路切换     │  │
│  │ 告警摘要       │   地图主体  │ 站点列表     │  │
│  │ 调度提示       │            │ 站点压力     │  │
│  └──────────────┘            └────────────┘  │
│                                              │
└──────────────────────────────────────────────┘
```

### 5.3 首页应展示的数据

#### 5.3.1 全局总览指标

建议保留或增强以下指标：

| 指标 | 含义 | 数据来源建议 |
| --- | --- | --- |
| 线路数量 | 当前系统运营线路数量 | `/api/public/routes` 或后台统计接口 |
| 今日客流 | 当前日期或选中日期范围总乘车记录数 | 统计接口聚合 |
| 最高满载率 | 当前日期范围内最高线路或班次满载率 | 满载率统计接口 |
| 告警站点数 | 高客流或拥挤站点数量 | 站点客流排行 + 阈值计算 |

#### 5.3.2 当前线路指标

切换线路后，左侧浮层应显示当前线路口径：

- 当前线路名称。
- 当前线路站点数。
- 当前线路日期范围内客流量。
- 当前线路平均或最高满载率。
- 当前线路热点站点。
- 当前线路状态：平稳 / 关注 / 拥挤。

#### 5.3.3 站点压力

站点可分为三类：

| 状态 | 颜色 | 判断建议 |
| --- | --- | --- |
| 高压力 | 红色 | 当前线路站点客流位于前 20%，或超过均值明显较多 |
| 正常偏高 | 黄色 | 当前线路站点客流位于中间区间 |
| 平稳 | 绿色 | 当前线路站点客流较低 |

注意：如果没有绝对真实阈值，MVP v2 可以使用“当前线路相对排名”判断站点压力。这种方式适合演示，也容易解释。

#### 5.3.4 告警摘要

首页告警不需要做复杂告警系统，建议先做“展示型告警摘要”。

告警类型可以包括：

- 高满载率告警：线路或班次满载率超过 85%。
- 热点站点告警：某站点上车人数在当前线路中排名靠前。
- 客流集中告警：早高峰或晚高峰客流明显高于平峰。
- 数据为空提示：选中日期范围没有统计数据。

### 5.4 建议实现步骤

#### 步骤 1：梳理首页现有数据流

需要检查：

- `frontend/src/views/HomeMap.vue`
- `frontend/src/components/RouteMap.vue`
- 前端统计 API 封装文件
- 后端公开统计接口

确认当前首页已经能拿到：

- 线路列表。
- 当前线路站点。
- 日期范围。
- 线路统计。
- 站点客流。
- 满载率或高峰统计。

#### 步骤 2：统一首页展示口径

首页需要区分两个口径：

1. 顶部总览：全线路口径。
2. 左侧浮层和右侧站点：当前选中线路口径。

避免出现“顶部指标和浮层指标重复但含义不同”的情况。

建议在代码中明确命名：

```text
globalOverview      全局总览
selectedRouteStats  当前线路统计
selectedStationFlow 当前线路站点客流
routeAlerts         当前线路告警摘要
```

#### 步骤 3：增强站点状态计算

前端可根据站点客流排行计算压力状态：

```text
如果站点客流 >= 当前线路站点最大客流的 70%：高压力
如果站点客流 >= 当前线路站点最大客流的 35%：正常偏高
否则：平稳
```

如果当前日期无数据，则所有站点显示为“暂无数据”或默认状态。

#### 步骤 4：优化地图标记和线路颜色

地图应体现交通语义：

- 红色：拥挤或高压力。
- 黄色：正常偏高。
- 绿色：平稳。
- 蓝色：默认线路或系统主色。

腾讯地图可用时：

- Marker 使用小图钉或圆点标记。
- 不同压力状态使用不同颜色。
- 点击站点显示站名、站序、客流量、压力状态。

腾讯地图 Key 缺失时：

- 降级线路图也要显示红黄绿状态。
- 不应出现空白首页。

#### 步骤 5：增加告警摘要卡片

左侧浮层中展示 2 到 4 条告警即可，不宜过多。

示例：

```text
高峰拥挤：公交3路早高峰平均满载率 91.2%
热点站点：五水商圈上车客流排名第 1
客流集中：早高峰客流约为平峰的 2.3 倍
```

#### 步骤 6：完成响应式检查

首页浮层和右侧面板容易遮挡地图，需要检查：

- 1366px 宽度下是否完整可用。
- 1920px 宽度下是否有大屏效果。
- 窄屏时右侧面板是否可以下移或折叠。

### 5.5 首页验收标准

- 打开首页后，第一眼能看出是公交运行态势平台。
- 顶部总览能展示全局线路、客流和告警信息。
- 切换线路后，左侧浮层和右侧站点面板能同步变化。
- 地图站点颜色能反映站点压力。
- 告警摘要能解释当前线路或当前日期范围的问题。
- 腾讯地图 Key 缺失时仍有可读的降级视图。
- 前端构建通过。

## 6. 功能优化点二：客流分析页增强

### 6.1 优化目标

客流分析页需要从“有图表”升级为“图表能说明问题”。

本页要回答答辩中常见问题：

- 哪条线路客流最高？
- 早晚高峰是否明显高于平峰？
- 哪些站点最拥挤？
- 当前线路是否需要增加班次？
- 最近一段时间客流是否稳定？

### 6.2 建议页面结构

```text
┌──────────────────────────────────────────────┐
│ 筛选区：线路 / 日期范围 / 快捷日期 / 查询按钮       │
├──────────────────────────────────────────────┤
│ 指标摘要：总客流 / 日均客流 / 高峰占比 / 热点站点    │
├──────────────────────┬───────────────────────┤
│ 日客流趋势折线图       │ 高峰/平峰对比柱状图       │
├──────────────────────┼───────────────────────┤
│ 线路对比图             │ 站点客流 TOP 排行         │
├──────────────────────────────────────────────┤
│ 图表解释区：自动生成 3-5 条分析结论                 │
└──────────────────────────────────────────────┘
```

### 6.3 建议增强图表

#### 6.3.1 日客流趋势图

用途：展示选中线路在日期范围内的客流变化。

建议展示：

- X 轴：日期。
- Y 轴：客流量。
- 折线：每日客流。
- 可选：平均线。

图表解释示例：

```text
选中日期范围内，公交3路日客流整体高于公交2路，工作日早晚高峰更明显。
```

#### 6.3.2 高峰/平峰对比图

用途：说明早高峰、晚高峰、平峰之间差异。

建议展示指标：

- 班均客流。
- 平均满载率。
- 总客流。

优先使用“班均客流”和“平均满载率”，因为只看总客流可能受班次数影响，答辩时不够严谨。

展示方式：

- 柱状图：早高峰、晚高峰、平峰。
- 每个柱子展示班均客流或平均满载率。
- 可在图表旁边显示一句解释。

#### 6.3.3 线路对比图

用途：展示 1/2/3 路之间的客流差异。

建议展示：

- 线路总客流对比。
- 线路平均满载率对比。
- 线路热点站点数量对比。

如果现有接口不支持一次获取所有线路统计，可以前端循环调用各线路统计接口后聚合。

#### 6.3.4 站点客流 TOP 排行

用途：展示热点站点。

建议展示：

- 站点名称。
- 上车人数。
- 占当前线路客流比例。
- 压力状态标签。

站点排行旁边可以加解释：

```text
热点站点集中在线路中段商圈和学校附近，说明该线路具有明显通勤和生活出行特征。
```

#### 6.3.5 自动图表解释区

无需调用大模型，前端或后端可根据统计结果生成规则化解释。

建议输出 3 到 5 条：

- 客流最高日期。
- 客流最高线路。
- 高峰和平峰差异。
- 热点站点。
- 是否达到调度关注阈值。

示例：

```text
1. 当前日期范围内，公交3路客流最高，适合作为重点调度线路。
2. 早高峰班均客流明显高于平峰，存在通勤集中出行特征。
3. 五水商圈、市一中等站点客流排名靠前，是主要客流压力点。
4. 若高峰平均满载率超过 85%，建议缩短发车间隔或增加车辆。
```

### 6.4 建议实现步骤

#### 步骤 1：确认现有统计接口能力

需要检查：

- 线路日客流接口。
- 站点客流排行接口。
- 高峰对比接口。
- 满载率接口。

如果接口已经满足展示需求，优先不新增接口。

#### 步骤 2：整理前端统计数据模型

建议在前端建立统一结构：

```text
routeFlowTrend       日客流趋势
peakComparison       高峰/平峰对比
stationRanking       站点排行
routeComparison      线路对比
analysisSummary      自动解释文本
```

#### 步骤 3：增加指标摘要卡片

建议摘要卡片包括：

- 总客流。
- 日均客流。
- 高峰客流占比。
- 平均满载率。
- TOP 站点。

#### 步骤 4：增强 ECharts 配置

图表需要注意：

- 标题清晰。
- tooltip 展示单位。
- 空数据时显示空状态。
- 图例不遮挡内容。
- 色彩语义统一：红黄绿蓝。

#### 步骤 5：实现自动解释

可在前端使用纯函数实现：

```text
generateFlowInsights(data) -> string[]
```

输入统计数据，输出分析结论列表。

#### 步骤 6：检查数据为空场景

如果日期范围没有数据，页面要提示：

- 当前日期范围暂无客流数据。
- 请重新生成模拟数据或切换日期范围。

### 6.5 客流分析页验收标准

- 图表不只是展示数据，还能解释数据。
- 能看出早高峰、晚高峰和平峰的差异。
- 能看出不同线路客流差异。
- 能看出热点站点。
- 空数据有明确提示。
- 页面适合截图放入答辩 PPT 或简历项目说明。

## 7. 功能优化点三：AI 助手报告化

### 7.1 优化目标

AI 助手不应只是普通聊天页面，而应输出“公交客流调度分析报告”。

报告需要包含：

- 关键指标。
- 异常判断。
- 调度建议。
- 风险提示。
- 数据依据。

### 7.2 建议页面结构

```text
┌──────────────────────────────────────────────┐
│ 分析条件：线路 / 日期范围 / 模型选择 / 问题输入      │
├──────────────────────────────────────────────┤
│ 报告摘要                                      │
├──────────────────────┬───────────────────────┤
│ 关键指标              │ 异常判断                │
├──────────────────────┼───────────────────────┤
│ 调度建议              │ 风险提示                │
├──────────────────────────────────────────────┤
│ 数据依据：统计时间、线路、客流、满载率、站点排行       │
└──────────────────────────────────────────────┘
```

### 7.3 AI 报告内容结构

建议 AI 输出统一为以下结构：

```text
1. 报告摘要
2. 关键指标
3. 异常判断
4. 调度建议
5. 风险提示
6. 数据依据
```

#### 7.3.1 报告摘要

说明本次分析对象和总体结论。

示例：

```text
本次分析对象为公交3路，统计区间为 2026-06-01 至 2026-06-07。该线路早晚高峰客流集中，部分站点存在较高上车压力，建议在高峰时段重点关注班次间隔。
```

#### 7.3.2 关键指标

必须包含真实统计数据，例如：

- 总客流量。
- 日均客流。
- 早高峰客流。
- 晚高峰客流。
- 平均满载率。
- 热点站点 TOP 3。

#### 7.3.3 异常判断

根据阈值判断：

- 满载率是否超过 85%。
- 高峰客流是否明显高于平峰。
- 是否存在站点客流过度集中。
- 日期范围是否无数据或数据偏少。

#### 7.3.4 调度建议

建议要具体，避免空泛：

- 是否增加班次。
- 是否缩短发车间隔。
- 是否关注某几个站点。
- 是否维持当前调度。
- 是否在平峰减少班次或延长间隔。

#### 7.3.5 风险提示

风险提示可以包括：

- 数据为模拟数据，仅用于课程演示。
- 当前统计区间较短，建议结合更长周期观察。
- 若缺少真实车辆定位，无法判断车辆实时拥堵位置。
- 若没有真实刷卡数据，站点压力为模拟客流估计。

#### 7.3.6 数据依据

AI 报告必须明确“依据哪些数据”。

示例：

```text
数据依据：线路日客流统计、站点上车客流排行、早晚高峰班均客流、班次满载率统计。
```

### 7.4 AI 输出约束

AI 不应：

- 声称使用了系统中不存在的数据。
- 声称接入了实时 GPS。
- 声称使用了真实刷卡支付数据。
- 在没有数据时编造具体数值。
- 给出与统计结果冲突的建议。

AI 应：

- 基于后端传入的统计数据回答。
- 明确说明分析对象和时间范围。
- 引用关键指标。
- 在数据不足时给出保守结论。
- 区分规则判断和 AI 语言解释。

### 7.5 建议实现步骤

#### 步骤 1：检查 AI 分析请求数据

需要确认 Java 后端传给 Python AI 服务的数据是否包含：

- 线路信息。
- 日期范围。
- 用户问题。
- 日客流统计。
- 高峰对比。
- 站点排行。
- 满载率统计。

如果已有字段不足，优先在后端聚合现有统计结果后再传给 AI 服务，不直接让 AI 服务查数据库。

#### 步骤 2：调整 AI 服务 Prompt 或模板

在 Python AI 服务中，将输出要求改成报告结构。

Prompt 应明确要求：

```text
请输出公交客流调度分析报告，必须包含：报告摘要、关键指标、异常判断、调度建议、风险提示、数据依据。
所有结论必须基于输入统计数据，不得编造不存在的数据。
```

#### 步骤 3：调整本地规则降级输出

未配置大模型 Key 或调用失败时，本地规则降级也要输出同样结构。

这样可以保证：

- 有 Key 时展示 AI 报告。
- 没 Key 时仍可演示报告化效果。
- 答辩时不依赖外部模型稳定性。

#### 步骤 4：前端按报告结构展示

前端不要只显示一整段文本，应拆分展示：

- 摘要卡片。
- 指标卡片。
- 异常判断列表。
- 调度建议列表。
- 风险提示列表。
- 数据依据区域。

如果后端暂时只返回纯文本，前端可先按标题切分；更推荐后端/AI 服务返回结构化字段。

#### 步骤 5：准备示例问题

页面可以内置 3 到 5 个示例问题，方便答辩演示：

- 最近一周哪条线路客流最高？
- 公交3路早高峰是否需要增加班次？
- 哪些站点客流压力最大？
- 晚高峰是否比平峰更拥挤？
- 当前线路是否存在调度风险？

### 7.6 AI 助手验收标准

- AI 输出像报告，不像普通聊天。
- 报告包含关键指标、异常判断、调度建议、风险提示和数据依据。
- 无 Key 或模型失败时仍能输出结构化降级报告。
- 报告中的数值来自统计数据。
- 页面适合截图展示“AI 调度分析能力”。

### 8.4 规则依据展示方式

建议将调度规则可视化展示。

```text

```

正常时段示例：

```text
当前平峰平均满载率为 24.3%，低于 30% 的低利用率阈值，因此系统建议适当延长发车间隔。
```

如果处于中间区间：

```text
当前高峰平均满载率为 72.5%，处于 60%-85% 观察区间，建议维持当前班次并继续观察热点站点变化。
```

### 8.5 建议级别语义

| 级别 | 展示颜色 | 含义 |
| --- | --- | --- |
| high / 高 | 红色 | 满载率高或客流压力大，需要优先处理 |
| medium / 中 | 黄色 | 存在关注点，建议持续观察 |
| low / 低 | 绿色 | 当前运行平稳或低利用率优化 |

注意：低级别不一定代表“不重要”，平峰低利用率也可以是优化方向。页面文案应解释清楚。

## 14. 风险与注意事项

### 14.1 数据口径风险

模拟数据不是官方真实客流数据，答辩时应表述为：

```text
线路和站点来自真实线路数据，运营客流为基于调查口径和业务规则生成的模拟数据，用于课程项目演示和统计分析验证。
```

不要说成真实刷卡数据。

### 14.2 AI 风险

AI 输出必须基于统计数据。若模型返回不稳定，应优先展示本地规则降级报告。

### 14.3 地图风险

腾讯地图 Key 可能受域名白名单、网络或额度影响。必须保留降级视图，确保答辩时不白屏。

### 14.4 范围风险

MVP v2 不应扩大到过多新功能。优先把现有功能讲清楚、展示好、解释明白。

## 16. 建议下一步

建议下一步不要一次性执行全部 MVP v2，而是按优先级逐步推进：

1. 先做首页态势展示增强。
2. 再做客流分析页增强。
3. 再做 AI 助手报告化。
