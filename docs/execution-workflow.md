# 项目任务执行规范 v1

## 1. 适用范围

本规范适用于本项目从阶段 1 到阶段 6 的所有开发任务。

阶段 7 由用户自行测试、验收、截图和整理最终报告素材；如果后续用户要求协助阶段 7，再按本规范执行对应修复或文档任务。

## 2. 单个任务执行闭环

每个任务开始前，必须按以下顺序执行：

1. 读取控制文档
   - `docs/rules.md`
   - `docs/status.md`
   - `docs/decisions.md`
   - `docs/next.md`
2. 理解当前任务
   - 明确当前只做哪一个任务。
   - 明确输入、输出、完成标准和非目标。
3. 阅读相关文档和代码
   - 文档优先读 `docs/requirements-v1.md`、`docs/mvp-v1.md`、`docs/database-design.md`、`docs/map-integration.md` 中与当前任务相关的部分。
   - 如果已有源码，必须先阅读相关源码再修改。
4. 给出执行计划
   - 简要说明本次准备修改哪些模块。
   - 说明会做哪些验证。
5. 小步修改
   - 一次只推进当前任务范围内的必要改动。
   - 不做无关重构和范围外优化。
6. 补测试或说明测试缺口
   - 有可运行测试时补充或执行测试。
   - 暂无测试框架时，说明采用的手动检查、脚本检查或缺口。
7. 运行检查
   - 根据任务类型运行相关检查，例如 JSON 校验、SQL 执行、后端测试、前端构建、AI 服务接口测试。
8. 自我 review
   - 检查是否偏离任务范围。
   - 检查是否违反既有决策。
   - 检查是否遗漏文档、测试或验收要求。
9. 更新文档
   - 每个任务结束时，只有项目状态发生实质变化才更新 `docs/status.md`。
   - 只有形成长期有效的业务或技术决策时才更新 `docs/decisions.md`。
   - 需要交接下一项明确任务时更新 `docs/next.md`。
   - 小任务不必频繁写入 `docs/log.md`。
10. 开始下一项任务
   - 只有当前任务通过检查后，才能进入下一项任务。

## 3. 阶段结束检查

每个大阶段结束后，必须进行一次阶段检查验证。

阶段检查必须回答：

- 本阶段计划任务是否全部完成。
- 本阶段产物是否能正常运行或被后续阶段使用。
- 是否存在阻塞下一阶段的问题。
- 运行了哪些检查，结果是什么。
- 有哪些测试缺口或遗留风险。
- 文档是否已同步更新。

阶段检查通过后，才进入下一阶段。

## 4. 轻量文档更新规则

实际开发中采用轻量记录方式，避免文档维护成本过高。

每个任务开始：

- 读取 `docs/rules.md`
- 读取 `docs/status.md`
- 读取 `docs/decisions.md`
- 读取 `docs/next.md`

每个任务结束：

- 只在状态有实质变化时更新 `docs/status.md`
- 形成长期决策时才更新 `docs/decisions.md`
- 需要明确下一步任务时更新 `docs/next.md`
- 小任务不必频繁写 `docs/log.md`

重要里程碑可在 `docs/log.md` 中用以下格式记录：

```text
## YYYY-MM-DD 阶段 X / 任务编号

- 任务：...
- 变更：...
- 检查：...
- 结果：通过 / 未通过
- 遗留：...
```

不重要的小改动不必写入 `docs/log.md`。

## 5. 对当前项目的评价

该执行流程是合理的，适合本项目。

原因：

- 本项目包含前端、Java 后端、Python AI 服务、MySQL、Redis、地图 SDK 和数据脚本，模块较多，必须通过控制文档避免上下文漂移。
- 每个阶段后做验证，可以防止问题累积到联调阶段才暴露。
- 每个任务都先读文档和相关代码，可以避免重复造轮子或破坏已有约定。
- 小步修改和自我 review 有利于控制开发范围，符合本项目“先做可运行 MVP”的目标。
