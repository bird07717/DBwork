# 城市公交客流分析与智能调度 Agent 平台需求文档 v1

## 1. 项目定位

本项目是一个面向城市公交场景的完整 Web 项目，目标是作为数据库课程项目和实习简历项目使用。系统围绕公交线路、站点、车辆、司机、班次和乘车记录进行数据管理，并结合地图展示、客流统计和 AI 分析能力，形成“交通领域 + 数据库设计 + 前后端开发 + AI 微服务”的综合项目。

项目第一版采用“游客免登录 + 管理员登录”的双身份模式。游客可以直接查看公交线路、地图和客流分析结果；管理员登录后负责基础数据维护、模拟数据生成、客流统计和调度建议管理。

## 2. 用户角色

### 2.1 游客

游客不需要登录，主要使用前台功能。

游客可执行的操作：

- 查看首页地图看板
- 查看公交线路列表
- 查看线路详情和站点顺序
- 查看线路在腾讯地图上的站点 Marker 和线路折线
- 查看公开客流趋势和站点客流统计
- 使用 AI 客流分析助手进行自然语言提问
- 查看系统生成的调度建议说明

游客不可执行的操作：

- 新增、修改、删除线路、站点、车辆、司机、班次等数据
- 生成模拟数据
- 访问管理员后台

### 2.2 管理员

管理员需要登录后台，负责项目网站和公交业务数据管理。

管理员可执行的操作：

- 登录管理员后台
- 管理公交线路
- 管理公交站点
- 管理线路与站点的关联关系
- 管理车辆信息
- 管理司机信息
- 管理班次信息
- 查看和维护乘车记录
- 生成或导入模拟运营数据
- 查看客流统计图表
- 生成 AI 调度建议
- 查看基础操作日志或数据状态

## 3. 用户使用路线

### 3.1 游客路线

```text
进入网站首页
→ 查看公交地图和线路概览
→ 选择一条公交线路
→ 查看线路详情、站点顺序和地图走向
→ 查看该线路客流趋势
→ 使用 AI 助手提出自然语言问题
→ 获取客流分析结果和调度建议说明
```

### 3.2 管理员路线

```text
进入管理员登录页
→ 输入账号密码登录
→ 进入后台首页
→ 维护线路、站点、车辆、司机等基础数据
→ 查看或生成班次和乘车记录
→ 查看客流统计图表
→ 使用 AI 生成调度建议
→ 检查系统数据状态
```

## 4. 功能需求

### 4.1 前台功能

#### 首页地图看板

- 展示 3 条公交线路的基础概览
- 接入腾讯地图 API
- 腾讯地图 Key 通过前端环境变量 `VITE_TENCENT_MAP_KEY` 读取
- 支持在地图中展示站点 Marker
- 支持绘制线路折线
- 支持选择不同线路并更新地图展示
- 展示线路数量、站点数量、当日客流等概览指标

#### 线路详情

- 展示线路基础信息
- 展示站点顺序
- 展示地图中的站点 Marker 和线路折线
- 点击站点后展示站点名称、所属区域、经纬度和客流概况

#### 客流分析

- 支持按线路查看客流趋势
- 支持查看早高峰、晚高峰和正常时段客流
- 支持查看站点客流排行
- 使用 ECharts 展示折线图、柱状图和排行图

#### AI 客流分析助手

- 支持游客输入自然语言问题
- 后端根据问题调用统计接口获取真实数据库结果
- AI 微服务基于统计数据生成自然语言分析报告
- 返回内容需要包含分析结论、关键数据和调度建议说明

### 4.2 管理员后台功能

#### 管理员登录

- 管理员使用账号密码登录
- 登录成功后进入后台
- 后台接口需要登录态校验
- 游客不能访问后台页面和后台接口

#### 线路管理

- 新增线路
- 修改线路
- 删除或停用线路
- 查询线路列表
- 维护线路编号、线路名称、起点、终点、方向、运营状态

#### 站点管理

- 新增站点
- 修改站点
- 删除或停用站点
- 查询站点列表
- 维护站点编号、站点名称、所属区域、经纬度

#### 线路站点管理

- 为线路配置站点
- 维护站点在线路中的顺序
- 支持查看某条线路经过的完整站点列表

#### 车辆管理

- 新增车辆
- 修改车辆
- 删除或停用车辆
- 查询车辆列表
- 维护车辆编号、车牌号、容量、状态

#### 司机管理

- 新增司机
- 修改司机
- 删除或停用司机
- 查询司机列表
- 维护司机姓名、工号、手机号、状态

#### 班次管理

- 新增班次
- 修改班次
- 查询班次列表
- 维护线路、车辆、司机、发车时间、班次状态
- 班次保存时需要保证车辆和司机引用有效

#### 乘车记录管理

- 查询乘车记录
- 按线路、站点、时间范围筛选乘车记录
- 查看上车站点、下车站点、乘车时间、支付方式
- 支持通过脚本或后台功能生成模拟乘车记录

#### AI 调度建议管理

- 管理员可以选择线路和时间范围生成调度建议
- 系统基于客流统计数据和规则模板生成建议
- AI 微服务将规则结果组织成可读的自然语言说明
- 调度建议可以保存到数据库中

### 4.3 第一版页面范围

第一版前端页面固定为 8 个页面，避免页面数量过多导致实现范围失控。

游客前台页面：

- 首页地图
- 线路详情
- 客流分析
- AI 助手

管理员后台页面：

- 管理员登录页
- 后台首页
- 基础数据管理
- 调度建议管理

基础数据管理页内部通过菜单或 Tabs 切换线路、站点、车辆、司机、班次和乘车记录模块，不为每个模块单独拆独立页面。

## 5. 数据需求

### 5.1 真实数据

用户提供 3 条真实公交线路及站点信息。系统第一版以这 3 条线路为核心展示对象。

真实线路数据建议包含：

- 线路编号
- 线路名称
- 起点站
- 终点站
- 站点顺序
- 站点名称
- 站点经纬度

### 5.2 模拟数据

除真实线路和站点外，其余运营数据模拟生成。

模拟数据规模：

- 线路：3 条，由用户提供
- 车辆：30 辆
- 司机：30 人
- 班次：每天约 100 条
- 乘车记录：30 天，约 1 万条

模拟数据需要支持：

- 早高峰客流
- 晚高峰客流
- 正常时段客流
- 不同线路的客流差异
- 不同站点的上车客流差异

## 6. AI 功能需求

AI 第一版只做两个核心能力：

- 自然语言客流分析
- 调度建议生成

AI 不直接凭空回答业务问题，而是通过后端统计接口获取数据库统计结果，再基于统计结果生成自然语言报告。

AI 分析流程：

```text
用户输入自然语言问题
→ Java 后端接收请求
→ Java 后端调用统计查询接口
→ Java 后端或 Python AI 服务整理结构化数据
→ Python FastAPI AI 微服务调用大模型
→ 返回分析结论和调度建议
```

示例问题：

- 最近一周哪条线路客流最高？
- 今天早高峰 1 路公交是否拥挤？
- 晚高峰是否需要增加班次？
- 正常时段哪条线路车辆利用率偏低？

## 7. 调度建议规则

第一版调度建议分两个模板。

### 7.1 早晚高峰模板

适用时段：

- 早高峰：07:00-09:00，核心高峰 07:30-08:30
- 晚高峰：17:00-19:00，核心高峰 17:30-18:30

关注指标：

- 满载率
- 客流峰值
- 站点候车压力
- 线路拥挤程度

建议逻辑：

- 当线路平均满载率大于等于 85% 时，建议增加班次或缩短发车间隔
- 当线路平均满载率在 60% 到 85% 之间时，建议维持当前班次并继续观察
- 当某些站点客流集中时，建议关注重点站点候车压力
- 当客流高于近 7 日均值 30% 时，提示可能存在异常客流

### 7.2 正常时段模板

适用时段：

- 非早晚高峰时段

关注指标：

- 平均客流
- 车辆利用率
- 班次密度

建议逻辑：

- 当满载率小于 30% 时，建议适当延长发车间隔或减少班次
- 当客流稳定时，建议维持当前班次
- 当车辆利用率偏低时，建议将部分车辆调配到高需求线路

## 8. 数据库需求

数据库使用 MySQL。

数据库课程重点体现：

- ER 图
- 主键
- 外键
- 索引
- 复杂 SQL 查询
- 事务

第一版暂不强制实现：

- 触发器
- 存储过程
- 视图

核心数据表：

- `admin_user` 管理员表
- `bus_route` 公交线路表
- `station` 站点表
- `route_station` 线路站点关联表
- `bus_vehicle` 车辆表
- `driver` 司机表
- `bus_schedule` 班次表
- `ride_record` 乘车记录表
- `dispatch_advice` 调度建议表
- `operation_log` 操作日志表

核心关系：

- 一条线路包含多个站点
- 一个站点可以属于多条线路
- 线路和站点通过 `route_station` 形成多对多关系
- 一条线路可以有多个班次
- 一个班次对应一辆车和一名司机
- 乘车记录关联线路、班次、上车站点和下车站点
- 调度建议基于线路、时段和客流统计结果生成

## 9. 技术选型

项目采用“本地运行核心代码 + Docker 运行数据库与基础设施”的方式。

核心代码本地运行：

- Vue 前端本地运行
- Java Spring Boot 后端本地运行
- Python FastAPI AI 微服务本地运行

基础设施 Docker 运行：

- MySQL 8.0
- Redis

技术栈：

- 前端：Vue 3、TypeScript、Vite、Element Plus、ECharts、腾讯地图 API
- Java 后端：Spring Boot、MyBatis Plus、Sa-Token、Knife4j
- AI 微服务：Python、FastAPI、大模型 API
- 数据库：MySQL 8.0
- 缓存：Redis
- 基础设施：Docker Compose
- 开发工具：IDEA、VS Code、PyCharm、Navicat 或 DataGrip、Git

地图接入细节见 `docs/map-integration.md`。

## 10. 非功能需求

数据完整性：

- 核心表需要主键
- 关键业务关系需要外键
- 编号类字段需要唯一约束
- 必填字段需要非空约束

查询效率：

- 客流统计需要支持时间范围查询
- `ride_record` 需要针对线路、站点和时间字段建立索引
- 统计接口需要避免一次性返回过大数据

安全性：

- 管理员后台必须登录
- 后台接口必须校验登录态
- 游客只能访问公开查询和 AI 分析接口

可维护性：

- 前后端分离
- Java 后端负责主业务
- Python AI 微服务负责 AI 分析
- 数据库脚本、模拟数据脚本和文档独立维护

可展示性：

- 首页必须有地图展示
- 客流分析必须有图表
- AI 分析结果必须能体现真实数据来源
- 后台管理功能需要能展示完整数据流转

## 11. 第一版边界

第一版不做：

- 真实车辆 GPS 定位
- 真实支付刷卡系统
- 复杂路径规划算法
- 多 Agent 协作系统
- 乘客实名注册
- 司机端 App
- 全量生产部署

第一版重点做：

- 3 条真实公交线路展示
- 腾讯地图线路和站点展示
- 模拟运营数据
- 游客前台查询
- 管理员后台维护
- 客流统计分析
- AI 自然语言客流分析
- AI 调度建议生成
