# 地图驱动前端优化计划

## 文档目的

本文档记录基于用户提供的前端模型图形成的页面优化方向、实施计划和验收标准。后续如果需要执行本任务，可将本文档作为前端改造的任务说明和设计依据。

当前阶段只记录方案，不直接修改前端实现代码。

## 模型图拆解

模型图整体适合作为本项目的主视觉参考，因为它同时覆盖了公交线路地图、客流指标、告警站点、客流分析图表、后台表格管理和 AI 调度助手，和本项目现有 MVP 页面范围高度一致。

### 可借鉴部分

- 地图作为首页第一视觉核心，公交线路以红、黄、绿状态色表达运行压力。
- 左上角信息浮层展示平台名称、实时状态、今日总客流、最高满载率和警报列表。
- 右侧线路面板展示站点顺序、当前站点状态、客流人数和调度建议。
- 客流分析页采用折线图、上下车对比和热力图表达不同时间维度的客流变化。
- 管理员后台采用侧边栏、顶部栏、表格主体的实用型管理布局。
- AI 页面同时包含分析报告和对话助手，适合表达“AI 调度决策辅助”。

### 不宜直接照搬部分

- 不应把所有信息堆到同一个页面，避免首页信息过载。
- 不应将通用 SaaS 模板风格直接覆盖游客端，游客端必须突出地图、线路和站点。
- 不应让 AI 助手页变成普通客服聊天页，应强调客流分析、风险提示和调度建议。
- 不应让后台管理页过度装饰，表格页应优先保证查询、筛选、编辑和状态展示效率。

## 总体设计方向

前端整体定位为“公交客流分析与调度决策平台”。游客端突出地图、线路、站点和客流状态；管理员端突出数据统计、基础数据管理和调度建议；AI 页面突出分析报告、关键发现和建议解释。

视觉关键词：

- 城市交通调度
- 地图驱动
- 数据分析
- 实用后台
- 红黄绿交通状态语义
- 浅色主界面，局部深色 AI 面板

建议色彩语义：

- 红色：高客流、拥挤、告警、需要处理
- 黄色：正常偏高、需要关注
- 绿色：平稳、低风险
- 蓝色：系统信息、筛选、主操作
- 深色：AI 助手或重点决策面板的局部强调

## 页面优化计划

### 第一阶段：首页地图驱动改造

优先改造 `frontend/src/views/HomeMap.vue` 和 `frontend/src/components/RouteMap.vue`。

目标是将首页升级为公交运行态势首页。用户进入系统后，应首先看到地图、线路、站点客流状态、今日总客流、最高满载率和警报站点。

建议布局：

- 主区域为地图，地图尽量占据首屏主要空间。
- 左上角悬浮信息面板展示平台名称、状态标签、今日总客流、最高满载率和警报列表。
- 右侧固定或悬浮线路面板展示线路选择、站点列表和当前站点状态。
- 地图上的线路和站点按客流强度显示红、黄、绿状态。
- 腾讯地图 Key 缺失时，继续保留现有降级线路图，但视觉上也应接近地图态势页。

建议交互：

- 支持切换线路。
- 支持点击站点查看客流状态。
- 支持告警列表点击后定位或高亮对应站点。
- 支持地图和右侧站点列表联动。

完成标准：

- 首页首屏不再像普通入口页，而是明确呈现公交运行态势。
- 今日总客流、最高满载率和警报站点可见。
- 地图站点颜色能反映客流强度。
- 右侧线路/站点面板可辅助理解地图。
- 地图 Key 缺失时仍有可看的降级视图。
- `npm run build` 通过。

### 第二阶段：线路详情页增强

优化 `frontend/src/views/RouteDetail.vue`。

目标是让线路详情页成为单条公交线路的运行档案。

建议布局：

- 顶部展示线路名称、站点数量、近 7 日客流、平均满载率等关键指标。
- 中间展示线路地图或降级路线图。
- 右侧或下方展示站点顺序、站点客流强度和当前选中站点详情。
- 下方展示近 7 日客流趋势和站点客流排行。

完成标准：

- 用户能清楚看出一条线路的基本情况、站点顺序和近期客流变化。
- 地图、站点列表和趋势图之间视觉层级清楚。

### 第三阶段：客流分析页视觉升级

优化 `frontend/src/views/FlowAnalysis.vue` 和 `frontend/src/components/ChartBox.vue`。

目标是让客流分析页更像模型图右上区域的数据分析面板。

建议保留三类核心图表：

- 客流时间序列：展示客流随日期或小时变化。
- 上下车或高峰对比：展示不同班次、时段或线路压力。
- 客流热力图：展示一周内不同时段的客流集中情况。

筛选器建议放在顶部，包括线路、日期范围和统计粒度。

完成标准：

- 用户能快速判断哪个时间段、哪条线路、哪些站点客流压力最大。
- 图表区域清晰，不堆叠过多无关指标。

### 第四阶段：AI 助手页重构

优化 `frontend/src/views/AiAssistant.vue`。

目标是从普通问答页升级为 AI 调度分析助手。

建议结构：

- 左侧或上方为问题输入、线路选择、日期范围和分析按钮。
- 主区域展示 AI 分析报告。
- 报告拆分为摘要、关键发现、风险提示、调度建议。
- 建议项使用高、中、低优先级标签。
- 对话输入可保留，但不作为唯一主体。

完成标准：

- AI 输出不只是长文本，而是可读的决策辅助报告。
- 调度建议的原因、风险和优先级清楚。

### 第五阶段：管理员后台视觉统一

优化以下文件：

- `frontend/src/views/admin/AdminDashboard.vue`
- `frontend/src/views/admin/AdminData.vue`
- `frontend/src/views/admin/AdminDispatch.vue`
- `frontend/src/views/admin/AdminLogin.vue`

目标是让后台保持实用型 SaaS 管理界面，同时统一到公交调度平台视觉。

建议：

- 后台首页使用统计卡片、快捷入口和近期状态。
- 基础数据管理页保持表格优先，强化搜索、筛选、状态标签和操作按钮。
- 调度建议管理页突出生成建议、建议结果和历史记录。
- 登录页保持简洁，适度体现公交调度系统主题。

完成标准：

- 后台页面信息密度合理，表格操作清晰。
- 管理员能快速完成查看、筛选、新增、编辑、停用和生成建议等操作。

## 实施边界

优先不修改数据库 schema。

优先不修改认证、权限和登录逻辑。

优先不引入新依赖，继续使用 Vue 3、TypeScript、Element Plus、ECharts、腾讯地图 JavaScript API GL。

优先使用现有后端接口和前端聚合数据。如果现有接口无法支撑首页概览指标，再单独评估是否新增后端概览接口。新增或改变公开 API 响应格式属于高风险变更，执行前应先确认。

## 建议执行顺序

1. 只执行首页地图驱动改造，确认整体视觉方向。
2. 执行线路详情页增强，让地图、站点和趋势图形成统一体验。
3. 执行客流分析页视觉升级，补齐图表分析感。
4. 执行 AI 助手页重构，强化调度决策辅助。
5. 执行管理员后台视觉统一，保证管理端实用性和一致性。

## 验证方式

每个阶段完成后至少执行：

```bash
cd frontend
npm run build
```

如具备浏览器环境，应人工检查：

- 首页地图是否正常显示。
- 腾讯地图 Key 缺失时降级视图是否可用。
- 不同屏幕宽度下信息面板是否遮挡地图或图表。
- 红、黄、绿状态色是否和业务含义一致。
- 图表文字、表格操作和按钮在移动端是否溢出。

## 剩余风险

- 模型图偏大屏监控和 SaaS Dashboard，移动端需要单独适配，不能直接压缩。
- 首页浮层和右侧面板容易遮挡地图，需要控制宽度、透明度和响应式折叠策略。
- 如果首页需要跨接口聚合较多数据，可能增加前端复杂度；必要时再评估后端概览接口。
- 深色 AI 面板与浅色主界面混用时，需要统一边框、阴影、字号和间距，避免拼贴感。
